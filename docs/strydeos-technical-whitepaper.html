1. The Blind Spot in Private Practice

Most private practice owners are clinicians first and business operators second. They know their clinical craft. They don't know their numbers.

The average UK physiotherapy practice owner cannot tell you their follow-up conversion rate, their programme compliance percentage, their utilisation gap, or how many patients dropped out mid-programme last quarter. They can't tell you because they've never seen these numbers connected into a single chain.

The data exists — scattered across practice management systems, exercise platforms, billing tools, and Google reviews. But no product unifies it. No product connects the dots from clinical performance through to revenue outcomes.

The revenue chain they don't know exists:

NPS → Google Reviews → Google ranking → new patient visibility → revenue. Clinic owners think reviews "just happen." StrydeOS systematically manufactures them from promoters.

StrydeOS exists because the founder runs a clinic and saw the gap firsthand. It is not a tech consultant's hypothesis. It is an operator's product, validated with real patient data at a real practice.

2. The Three-Stakeholder Model

Every decision in StrydeOS is evaluated against three stakeholders simultaneously. No competitor addresses all three. Most address one.

┌──────────────┐ │ Happy Physio │ └──────┬───────┘ │ ┌───────────┴───────────┐ │ │ ┌───┴──────────┐ ┌────────┴───────┐ │ Happy Patient │ │ Happy Owner │ └──────────────┘ └────────────────┘
  • Clinician — Clarity on their own performance. Not blame; coaching. They see their follow-up rate, programme compliance, and patient outcomes compared to benchmarks. The tone is supportive, not punitive.
  • Patient — Better outcomes, better follow-through. Timely reminders, exercise programmes, continuity between sessions. They feel looked after.
  • Owner — EBITDA, reputation, retention. Every metric connects to revenue. No vanity stats.

Design principle: If a metric doesn't connect to outcomes or revenue, it doesn't belong in the product. NPS and Google Reviews are treated as EBITDA levers, not feedback widgets.

Ava — AI Voice Receptionist

Ava answers inbound patient calls, understands their request, and takes action. She books appointments directly into the clinic's PMS, handles rescheduling, and triages urgent requests. If a caller has a complaint or needs human help, Ava warm-transfers the live call to reception — no dropped calls, no callback promises.

  • Voice engine: ElevenLabs Conversational AI (sub-400ms latency, natural turn-taking)
  • Telephony: Twilio SIP trunking with dedicated UK phone numbers per clinic
  • Conversation routing: LangGraph state machine for structured call flows
  • Warm transfers: Complaint detection → live Twilio redirect to reception via TwiML
  • Phone provisioning: One-click UK number purchase + SIP trunk + agent creation
  • Knowledge base: Per-clinic CRUD editor — clinic hours, services, insurance, policies
  • PMS write-back: Appointments created via n8n automation with automatic retry on failure
  • Guardrails: Insurance/billing advice blocked — redirects to reception

Intelligence — Clinical Performance Dashboard

The owner's command centre. Six validated KPIs tracked per clinician, per week, with rolling 90-day windows. Context-aware insights that explain why numbers moved, not just what they are.

Pulse — Patient Retention Engine

Automates every touchpoint between sessions. Risk scoring identifies patients likely to drop off. Communication sequences (SMS, email) re-engage them before they disengage.

  • Risk scoring: Multi-factor model using session gaps, programme adherence, outcome trajectories
  • Lifecycle states: Active, at-risk, lapsed, discharged — with automated transitions
  • Comms sequences: Configurable multi-step campaigns triggered by lifecycle events
  • Heidi complexity layer: Clinical documentation AI enriches patient profiles with complexity signals, ICD-10 codes, and session notes
  • Clinician engagement: Per-clinician digest opt-outs, engagement toggles, observational notes on insights

Multi-Tenant Data Model

Every query, every document, every metric is partitioned by clinicId. This is not a filter — it is the fundamental isolation strategy. A clinic can never see another clinic's data, not through the UI, not through the API, not through a query bug. 

// Firestore structure
clinics/{clinicId}
  /appointments/{appointmentId}
  /clinicians/{clinicianId}
  /voiceInteractions/{callId}
  /metrics_weekly/{weekKey}
  /sequence_definitions/{seqId}
  /comms_log/{logId}

users/{uid}
  .clinicId  → points to the clinic they belong to
  .role      → superadmin | owner | admin | clinician

Resilience Architecture

Integration failures are isolated per-clinic using Promise.allSettled. If one clinic's PMS goes down, every other clinic continues syncing normally. Failed syncs auto-retry on the next 4-hour cron cycle. No manual intervention required.

Clinic A (WriteUpp) → Real-time webhook (primary) → 4h cron (fallback) Clinic B (Cliniko) → 4h cron (only path) Clinic C (Zanda) → 4h cron (only path) If Cliniko API fails at 10:00: • Clinic B sync fails → logged to Sentry with clinicId tag • Clinics A, C continue syncing normally • Clinic B auto-retries at 14:00 • No manual intervention required

Integration partner uptime: Zanda UK maintains 99.99% uptime (90-day average) — matching AWS S3. Cliniko reports 99.98%. WriteUpp uses Azure geo-replication with 8 geographically distributed real-time copies and is the only PMS with real-time webhook support.

6. Data Pipeline

The sync pipeline runs on a 4-hour cron cycle (with real-time webhook overlay for WriteUpp). Each cycle pulls appointments, patients, clinicians, and HEP data from connected systems, normalises it through per-PMS adapters, and writes to Firestore.

PMS APIs Adapters Firestore ───────── ────────── ────────── WriteUpp ─┐ ┌─ appointments Cliniko ─┤──→ Normalised Schema ──┤─ clinicians Halaxy ─┤ ├─ patients Zanda ─┘ └─ metrics_weekly HEP APIs ───────── Physitrack ──→ Programme/Compliance ──→ physitrack_programs Clinical ───────── Heidi Health ──→ Complexity Signals ──→ patient profiles

Metrics Computation

KPIs are computed and cached in metrics_weekly — never derived on-the-fly from raw appointment data. This keeps dashboard loads fast regardless of data volume. Each metric includes both weekly snapshots and rolling 90-day windows.

CSV Fallback

Every PMS type supports CSV import as a manual fallback: import-{clinicId}@ingest.strydeos.com. This covers PMS providers without API access (TM3, Jane App) and serves as a disaster recovery path if API integrations fail.

  • Data residency: All data stored in europe-west2 (London). Never leaves the UK.
  • Encryption: AES-256 at rest (Firestore default), TLS 1.3 in transit.
  • Authentication: Firebase Auth with MFA (TOTP and phone), session cookies (HMAC-signed, HttpOnly, 1hr TTL).
  • RBAC: Enterprise 4-tier hierarchy enforced at middleware, client, API, and database layers.
  • Webhook verification: ElevenLabs (HMAC-SHA256), Twilio (request signature), Stripe (webhook secret).
  • Rate limiting: Applied to all API routes. Aggressive limits on cost-sensitive operations (phone provisioning: 2/min).
  • Secret detection: Pre-commit hook scans for credentials before code is committed.
  • SAR compliance: Subject Access Request templates (acknowledgement, completion, refusal) for GDPR data subject rights.
  • Audit logging: Integration health tracking per-clinic, per-provider, with stage-level stats.
  • Cookie consent: GDPR-compliant banner with genuine accept/decline functionality.

Toll fraud prevention: Ava's call transfer system looks up reception phone numbers from Firestore — never from query parameters. This prevents attackers from redirecting calls to premium-rate numbers.

10. Scalability & Expansion Path

Multi-Tenant Architecture

clinicId partitioning is baked into every query, every Firestore rule, every API route. Adding a new clinic is a document write — not a deployment. The pipeline already handles multiple clinics in parallel with per-clinic error isolation.

Vertical Expansion

The three-stakeholder model and integration-first architecture are portable across allied health verticals:

  1. Physiotherapy (live — Spires pilot)
  2. Chiropractic / Osteopathy / Sports Therapy — same PMS ecosystem, same KPIs
  3. Medspa / Aesthetics — Pabau PMS, different KPIs (treatment completion, course rebooking)
  4. Dental — Dentally, SOE; hygienist utilisation, treatment plan acceptance, recall rate

Each vertical needs: recalibrated KPI set + PMS integration + landing page. The core platform is unchanged.

Geographic Expansion

UK first (2–3 clinics stable) → Australia (similar private practice structure) → Canada → US. Firestore's multi-region support and per-clinic isolation make geographic expansion a configuration change, not an architecture change.

White-Label Path

Future: per-clinic branding (logo, colours), Vapify layer for white-label Ava (wraps ElevenLabs at reseller phase), and per-seat SaaS pricing for larger groups.
